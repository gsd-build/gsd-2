3. In the `session_update` handler (after `applyState(updated)` at L453), add fork message copy logic:
```typescript
// Fork message copy: if we just created a fork, copy parent messages to new session
if (pendingForkRef.current) {
  const forkSourceId = pendingForkRef.current;
  pendingForkRef.current = null;
  // Find the NEW session: present in tabs but not in stateRef.current.sessions
  const previousIds = new Set(stateRef.current.sessions.map(s => s.id));
  const newSession = tabs.find(t => !previousIds.has(t.id));
  if (newSession) {
    setMessagesBySession((prev) => {
      const newMap = new Map(prev);
      const parentMsgs = prev.get(forkSourceId) ?? [];
      // Copy parent messages as display context (shallow copy)
      newMap.set(newSession.id, [...parentMsgs]);
      return newMap;
    });
  }
}
```

IMPORTANT: The `previousIds` comparison must use `stateRef.current.sessions` (the BEFORE state) not `sessions` (React state, which may already be updated by applyState). Since `applyState` calls `setSessions` which is async, `stateRef.current` still has the old value at this point.

Wait — `stateRef.current` is updated in a useEffect that depends on `sessions`, so it reflects the previous render's state. This means `stateRef.current.sessions` has the PRE-update sessions. The `tabs` variable has the POST-update sessions from the server. So `tabs.find(t => !previousIds.has(t.id))` correctly finds the NEW session. This is correct.
