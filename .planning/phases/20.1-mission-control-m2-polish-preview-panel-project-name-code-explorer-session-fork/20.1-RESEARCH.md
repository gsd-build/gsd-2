This phase is a polish and bug-fix pass on an already-shipped React 19 + Bun application. The codebase is mature and fully readable — all source files are available. Research is therefore codebase-forensic: understand what already exists, confirm what is broken, document the exact extension points.

The 10 items fall into five categories: (1) React error boundary wrapping for iframe crash isolation, (2) preview panel architecture refactor for multi-server + multi-viewport support, (3) Code Explorer modal — the one genuinely new component, requiring either Monaco or CodeMirror as an editor plus server-side file read/write API extensions, (4) session management fixes (fork message copy, refresh recovery banner), and (5) surface-level UI tweaks (logo swap, project name, scrollbar utilities). **Primary recommendation:** Build in the plan order: error boundary first (unblocks stable preview), then multi-server architecture (replaces broken single-port model), then Code Explorer (largest new component), then session fork + refresh recovery, then the surface fixes last.

### Pattern 1: React Error Boundary (Class Component)

React 19 error boundaries still require class components (no hook equivalent).

**What:** Wrap each `DeviceFrame` in an `ErrorBoundaryFrame` class component that renders a fallback on uncaught render error.

**When to use:** Around any iframe-containing component where a rendering exception would otherwise propagate up the React tree and freeze the whole app.

```typescript
// Source: React docs + Phase 11.1 precedent in error-boundary.test.tsx
import { Component, type ReactNode, type ErrorInfo } from "react";

interface State { hasError: boolean; error?: Error }

export class ErrorBoundaryFrame extends Component<
  { children: ReactNode; fallback?: ReactNode },
  State
> {
  static getDerivedStateFromError(error: Error): State {
    return { hasError: true, error };
  }
  state: State = { hasError: false };

  componentDidCatch(error: Error, info: ErrorInfo) {
    console.error("[ErrorBoundaryFrame] iframe error:", error, info);
  }

  render() {
    if (this.state.hasError) {
      return (
        this.props.fallback ?? (
          <div className="flex items-center justify-center h-full text-slate-500 text-xs font-mono p-4">
            Preview unavailable
          </div>
        )
      );
    }
    return this.props.children;
  }
}
```

**IMPORTANT — React 19 renderToString note (from Phase 11.1 decisions):**
> React 19 renderToString no longer triggers error boundary lifecycle — error-boundary test updated to direct class instance testing (getDerivedStateFromError + render())

Tests for this boundary must use direct class instance testing, NOT `renderToString`.

### Pattern 2: Port Scan via Parallel Fetch

**What:** Attempt `fetch` to each candidate port simultaneously; first success wins and becomes active port.

```typescript
// Parallel fetch race — fastest to respond wins
const CANDIDATE_PORTS = [3000, 4173, 5173, 8080, 8000];

async function scanForDevServers(): Promise<{ port: number; type: "frontend" | "backend" }[]> {
  const results = await Promise.allSettled(
    CANDIDATE_PORTS.map(async (port) => {
      const res = await fetch(`http://localhost:${port}/`, {
        signal: AbortSignal.timeout(800), // fast timeout
        mode: "no-cors", // avoid CORS failure masking a real hit
      });
      // Content-type heuristic: text/html = frontend, application/json = backend
      const ct = res.headers.get("content-type") ?? "";
      const type = ct.includes("html") ? "frontend" : "backend";
      return { port, type };
    })
  );
  return results
    .filter((r) => r.status === "fulfilled")
    .map((r) => (r as PromiseFulfilledResult<{ port: number; type: "frontend" | "backend" }>).value);
}
```

**CAVEAT:** `mode: "no-cors"` means `res.headers` will be empty for opaque responses — the content-type heuristic only works for same-origin or CORS-enabled servers. For localhost this is generally fine. Use port number heuristic as fallback (5173 = Vite = frontend, 8000/8080 = backend typical).

import { EditorView, basicSetup } from "codemirror";
import { javascript } from "@codemirror/lang-javascript";
import { EditorState } from "@codemirror/state";

export function CodeMirrorEditor({

  language,
  onChange,
  readOnly = false,

  language: string;

  readOnly?: boolean;

  useEffect(() => {

### Pattern 4: Server-Side File Read/Write (extending fs-api.ts)

The existing `fs-api.ts` exports `validatePath()` and `listDirectory()`. The Code Explorer needs two more routes:

```typescript
// GET /api/fs/read?path=X — read file content
// POST /api/fs/write — write file content (body: { path: string, content: string })
```

**Security:** Both routes MUST call `validatePath(requestedPath, projectRoot)` where `projectRoot` is the current project directory (same boundary as `detectBoundaryViolation`). The project root must be sourced from the pipeline's `repoRoot` state.

The server needs access to the current `repoRoot`. The cleanest pattern is:
- Add `getProjectRoot()` to `PipelineHandle`
- Pass it to the HTTP handler in `server.ts`
- `handleFsRequest` receives an optional `allowedRoot` argument for the read/write routes

### Pattern 5: Fork Message Copy

**Current state (from session-manager.ts L98-103):**
```typescript
if (opts?.forkFromSessionId) {
  const source = this.sessions.get(opts.forkFromSessionId);
  if (source) {
    claudeSessionId = source.processManager.sessionId ?? source.claudeSessionId;
  }
}
```

The fork currently copies the Claude session ID (for `--resume`) but does NOT copy the message history. The fix lives in `useSessionManager.ts` client-side:

**Fix location:** `createSessionAction` in `useSessionManager.ts` — after `session_create` is dispatched and a `session_update` response arrives, pre-populate `messagesBySession` for the new session with a copy of the fork source's messages.

The server sends back a `session_update` with the new session ID. The client must:
1. Know which session was the fork source (track `pendingForkFrom` ref)
2. On receipt of `session_update` that includes a new session ID: copy parent messages into it

```typescript
// In useSessionManager — createSessionAction
const pendingForkRef = useRef<string | null>(null);

const createSessionAction = useCallback((forkFrom?: string) => {
  if (forkFrom) pendingForkRef.current = forkFrom;
  const payload: Record<string, unknown> = { type: "session_create" };
  if (forkFrom) payload.forkFromSessionId = forkFrom;
  send(JSON.stringify(payload));
}, [send]);

// In handleMessage — session_update handler
if (msgType === "session_update") {
  // ... existing code ...
  // After applyState, if pendingForkRef is set:
  if (pendingForkRef.current) {
    const forkSourceId = pendingForkRef.current;
    pendingForkRef.current = null;
    // Find the newly added session (not in previous sessions list)
    const newSession = tabs.find((t) => !stateRef.current.sessions.some((s) => s.id === t.id));
    if (newSession) {
      setMessagesBySession((prev) => {
        const newMap = new Map(prev);
        const parentMsgs = prev.get(forkSourceId) ?? [];
        // Copy parent messages as display context (mark them as forked)
        newMap.set(newSession.id, [...parentMsgs]);
        return newMap;
      });
    }
  }
}
```

### Pattern 6: Reconnect Recovery Banner

**Current state:** `useReconnectingWebSocket` fires `onReconnect` callback on reconnect. `useSessionManager` already calls `send(session_list)` on reconnect. Sessions with `processing: true` (yellow dot) but no active stream stay stuck.

**Fix:** After reconnect + session_list response, if a session has `isProcessing: true`, start a 3-second timer. If no `chat_event` arrives for that session within 3 seconds, set a `stuckSessionId` state → renders the reconnect banner.

```typescript
// In useSessionManager — after session_update handler
const stuckTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null);
const [stuckSessionId, setStuckSessionId] = useState<string | null>(null);

// On session_update after reconnect:
const processingSession = tabs.find((t) => t.isProcessing);
if (processingSession && isReconnected) {
  stuckTimerRef.current = setTimeout(() => {
    setStuckSessionId(processingSession.id);
  }, 3000);
}

// On chat_event arrival: clear the timer
if (stuckTimerRef.current) {
  clearTimeout(stuckTimerRef.current);
  stuckTimerRef.current = null;
  setStuckSessionId(null);
}
```

The banner component is a non-intrusive `position: fixed; bottom: 16px` strip (not a full-screen overlay).

### Anti-Patterns to Avoid

- **Adding CodeMirror as a dev dependency only:** It must be in `dependencies` — it ships in the browser bundle.
- **Using `mode: "cors"` in port scan:** Will silently fail for servers without CORS headers. Use `mode: "no-cors"` or wrap in try/catch.
- **Adding Code Explorer as a new ViewType:** The decision is explicit — it is a modal overlay, not a route. Do not add `"code-explorer"` to `ViewType`.
- **Storing fork messages server-side:** Fork message copy is client-side display state only. Do not serialize it to session metadata.
- **Using window.location.reload() for reconnect:** The decision is explicit — user must click Reconnect manually.

## Common Pitfalls

### Pitfall 1: React 19 Error Boundary Testing
**What goes wrong:** Test using `renderToString` or React Testing Library expects error boundary to catch during render — fails silently in React 19.
**Why it happens:** React 19 changed `renderToString` behavior for error boundaries.
**How to avoid:** Use direct class instance testing: `const boundary = new ErrorBoundaryFrame({}); const state = ErrorBoundaryFrame.getDerivedStateFromError(new Error("test")); expect(state.hasError).toBe(true);`
**Warning signs:** Test passes in isolation but the boundary doesn't actually intercept errors in the browser.

### Pitfall 2: CodeMirror 6 + Bun Bundler
**What goes wrong:** Bun may not resolve `@codemirror/*` ESM imports if package.json `exports` fields use conditional `browser` vs `node` variants incorrectly.
**Why it happens:** CodeMirror 6 packages use `exports` fields — Bun respects them but some older @codemirror packages may have issues.
**How to avoid:** Use `bun add` (not `npm install`). If bundling fails, add the problematic package to `bun build --external` and load via dynamic import from a CDN in dev, or vendor the dist.
**Warning signs:** `Cannot find module '@codemirror/view'` at build time despite being in node_modules.

### Pitfall 3: Port Scan CORS / Opaque Responses
**What goes wrong:** All port scan `fetch` calls resolve as "successful" even when nothing is listening, because `mode: "no-cors"` returns opaque responses.
**Why it happens:** `fetch` with `mode: "no-cors"` doesn't throw for CORS failures — it returns an opaque 0-status response.
**How to avoid:** Use `AbortSignal.timeout(800)` — a real server responds; nothing listening causes a connection refused error (which `Promise.allSettled` catches as rejected). Treat rejected = no server.
**Warning signs:** All 5 ports "found" simultaneously on a machine where no dev server is running.

### Pitfall 4: Multi-Server State Shape Complexity
**What goes wrong:** `usePreview` currently tracks a single `port: number | null`. Extending to multi-server requires a new state shape; if not planned carefully it breaks the existing `AppShell` → `PreviewPanelWithState` contract.
**Why it happens:** Single-port → multi-server is an interface-breaking change.
**How to avoid:** Keep backward compatibility: `port` prop becomes `servers: DetectedServer[]` + `activeFrontendPort: number | null`. The `PreviewPanel` consuming component is fully rewritten — it already has a clean props contract.
**Warning signs:** TypeScript errors in AppShell after changing usePreview return type.

### Pitfall 5: SVG Logo Already Exists
**What goes wrong:** Implementer creates a text fallback when the SVG already exists in `/assets/`.
**Why it happens:** The CONTEXT.md says "if no SVG asset exists" — easy to miss.
**How to avoid:** The SVG `gsd-logo-2000-transparent.svg` is already at `public/assets/gsd-logo-2000-transparent.svg`. The fix is a one-line `src` swap in `GsdLogo.tsx`. No text fallback needed.

### Pitfall 6: Session Fork Timing
**What goes wrong:** `pendingForkRef` gets consumed by a session_update that was triggered by a different event (e.g., session rename), copying messages to the wrong session.
**Why it happens:** WebSocket events can arrive out of order or batched.
**How to avoid:** Only consume `pendingForkRef` when the session_update adds a NEW session (not present in previous `stateRef.current.sessions`). If session count doesn't increase, don't consume.

### Pitfall 7: File Write API Security
**What goes wrong:** Code Explorer write endpoint allows writes to files outside project root if `allowedRoot` validation is omitted.
**Why it happens:** `validatePath` in `fs-api.ts` has the `allowedRoot` parameter optional — easy to call without it.
**How to avoid:** The write route MUST pass `projectRoot` as `allowedRoot`. Return HTTP 403 if path escapes.

## Open Questions

1. **CodeMirror 6 Bun bundling verification**
   - What we know: Bun can bundle ESM packages; CodeMirror 6 is ESM-first
   - What's unclear: Whether `bun-plugin-tailwind` + multiple `@codemirror/*` packages coexist without conflict in the existing build config
   - Recommendation: The Wave 0 plan should include a standalone bundleability test (`bun build src/frontend.tsx`) after adding CodeMirror. If it fails, fallback to a textarea + highlight.js approach.

2. **Project root availability in fs-api.ts write route**
   - What we know: `repoRoot` is a closure variable in `pipeline.ts`; `handleFsRequest` is a standalone function that doesn't currently receive it
   - What's unclear: Cleanest injection pattern without breaking existing tests
   - Recommendation: Add an optional `allowedRoot?: string` parameter to `handleFsRequest`, sourced from `pipeline.getProjectRoot()` in `server.ts`. Existing callers without `allowedRoot` continue to work for read-only routes.

3. **Dev server content-type detection accuracy**
   - What we know: `mode: "no-cors"` returns opaque responses with no headers
   - What's unclear: Whether any common dev servers (Vite, Next.js) send CORS headers on their root responses
   - Recommendation: Use connection-refused-vs-success as the primary signal. Use port number heuristics as secondary (5173 = Vite = frontend; 3000 could be either). Document that "frontend vs backend" classification is best-effort.
