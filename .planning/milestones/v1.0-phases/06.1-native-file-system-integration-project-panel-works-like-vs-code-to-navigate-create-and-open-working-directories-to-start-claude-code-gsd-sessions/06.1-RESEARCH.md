# Phase 06.1: Native File System Integration - Research

**Researched:** 2026-03-10
**Domain:** File system browsing, project directory management, Bun HTTP APIs, React tree components
**Confidence:** HIGH

## Summary

Phase 06.1 adds a VS Code-like project panel that lets users browse the local file system, select or create working directories, and open them as GSD/Claude Code sessions. Currently, the app hardcodes a single project directory (the repo root resolved in `server.ts`), and the sidebar's `ProjectList` component only shows the current project from `.planning/STATE.md`. There is no way to switch projects, browse directories, or start new sessions from within the UI.

The implementation requires two layers: (1) server-side REST API endpoints on the existing Bun HTTP server for directory listing, directory creation, `.planning/` detection, and project switching; (2) a React file tree component in the sidebar that replaces the current static `ProjectList` with an interactive directory browser. When a user selects a directory, the server must re-point the file watcher, state deriver, and Claude process manager to the new working directory.

**Primary recommendation:** Add REST endpoints to the existing `Bun.serve()` in `server.ts` for file system operations. Build a custom recursive tree component (no external dependency needed given the project's existing pattern of hand-built components with Tailwind + lucide-react). Wire project switching through WebSocket to trigger pipeline reconfiguration.

### Pattern 1: Server-Side File System API
**What:** REST endpoints on the existing Bun HTTP server for directory browsing
**When to use:** Whenever the frontend needs file system access
**Example:**
```typescript
// In server.ts fetch handler, add route matching before the 404 fallback
async function handleFsRequest(req: Request): Promise<Response | null> {
  const url = new URL(req.url);

  if (url.pathname === "/api/fs/list" && req.method === "GET") {
    const dirPath = url.searchParams.get("path") || os.homedir();
    const entries = await readdir(dirPath, { withFileTypes: true });
    const items = entries
      .filter(e => !e.name.startsWith(".") || e.name === ".planning")
      .map(e => ({
        name: e.name,
        path: join(dirPath, e.name),
        isDirectory: e.isDirectory(),
        hasPlanning: false, // checked separately
      }));
    return Response.json(items);
  }

  if (url.pathname === "/api/fs/detect-project" && req.method === "GET") {
    const dirPath = url.searchParams.get("path")!;
    const hasPlanningDir = await access(join(dirPath, ".planning"))
      .then(() => true).catch(() => false);
    return Response.json({ isGsdProject: hasPlanningDir, path: dirPath });
  }

  if (url.pathname === "/api/fs/mkdir" && req.method === "POST") {
    const { path: newDir } = await req.json();
    await mkdir(newDir, { recursive: true });
    return Response.json({ created: true, path: newDir });
  }

  if (url.pathname === "/api/project/switch" && req.method === "POST") {
    const { path: projectPath } = await req.json();
    // Triggers pipeline reconfiguration
    return Response.json({ switched: true, path: projectPath });
  }

  return null; // Not an API route
}
```

### Pattern 2: Pipeline Hot-Switch
**What:** Reconfigure the entire file-to-state pipeline when user switches projects
**When to use:** When a directory is selected as the active project
**Example:**
```typescript
// Extend PipelineHandle with a switchProject method
export interface PipelineHandle {
  stop(): void;
  processManager: ClaudeProcessManager;
  switchProject(newPlanningDir: string): Promise<void>;
}

// Implementation: stop watcher, rebuild state, restart watcher, reset Claude CWD
async switchProject(newPlanningDir: string): Promise<void> {
  watcher.close();
  const repoRoot = resolve(newPlanningDir, "..");
  currentState = await buildFullState(newPlanningDir);
  // Broadcast full state to all clients
  wsServer.broadcast({ type: "full", state: currentState, ... });
  // Kill existing Claude process, create new one with new CWD
  await processManager.kill();
  processManager = new ClaudeProcessManager(repoRoot);
  processManager.start();
  // Restart watcher on new directory
  watcher = createFileWatcher({ planningDir: newPlanningDir, ... });
}
```

### Pattern 3: Recursive Tree Component
**What:** A self-rendering tree node that lazily loads children on expand
**When to use:** For the directory browser UI
**Example:**
```typescript
interface FileTreeNodeProps {
  entry: FileSystemEntry;
  depth: number;
  onSelect: (path: string) => void;
}

function FileTreeNode({ entry, depth, onSelect }: FileTreeNodeProps) {
  const [expanded, setExpanded] = useState(false);
  const [children, setChildren] = useState<FileSystemEntry[] | null>(null);

  const handleToggle = async () => {
    if (!entry.isDirectory) return;
    if (!expanded && children === null) {
      const res = await fetch(`/api/fs/list?path=${encodeURIComponent(entry.path)}`);
      setChildren(await res.json());
    }
    setExpanded(!expanded);
  };

  return (
    <div>
      <button
        onClick={() => entry.isDirectory ? handleToggle() : onSelect(entry.path)}
        style={{ paddingLeft: `${depth * 16 + 8}px` }}
        className="flex items-center gap-2 w-full text-left py-1 px-2 hover:bg-navy-700 text-slate-300 text-xs font-mono"
      >
        {entry.isDirectory ? (
          expanded ? <ChevronDown size={12} /> : <ChevronRight size={12} />
        ) : <File size={12} />}
        {entry.isDirectory && entry.hasPlanning && (
          <span className="text-cyan-accent">GSD</span>
        )}
        <span className="truncate">{entry.name}</span>
      </button>
      {expanded && children?.map(child => (
        <FileTreeNode key={child.path} entry={child} depth={depth + 1} onSelect={onSelect} />
      ))}
    </div>
  );
}
```

### Anti-Patterns to Avoid
- **Eagerly loading the entire file tree:** Load children lazily on expand. Even a modest home directory has thousands of entries.
- **Exposing arbitrary file system access without path validation:** Sanitize paths server-side. Reject paths containing `..` traversal or absolute paths outside allowed roots.
- **Blocking the Bun event loop with sync fs calls:** Use `readdir` (async) not `readdirSync` in API handlers.
- **Storing project list in the React state tree only:** Persist recent projects to a JSON file (e.g., `~/.gsd/recent-projects.json`) so they survive page refreshes and server restarts.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Path traversal security | Custom string checking | `path.resolve()` + allowlist check | Path normalization is tricky on Windows with mixed separators |
| Cross-platform path separators | Manual replace | `path.join()` / `path.normalize()` | Already handles Windows backslashes vs Unix forward slashes |
| File type detection | stat + manual checks | `readdir({ withFileTypes: true })` | Returns Dirent objects with `isDirectory()`, `isFile()` etc. |

**Key insight:** The file system operations here are straightforward Node.js `fs` calls. The complexity is in the UI state management (expand/collapse, lazy loading, selected state) and in the pipeline hot-switch when changing projects.

## Common Pitfalls

### Pitfall 1: Windows Path Handling
**What goes wrong:** Paths use backslashes on Windows but URLs and JSON typically use forward slashes. Mixing causes path resolution failures.
**Why it happens:** The server runs on Windows (confirmed in project context). `path.join()` produces backslashes, but URL query params and JSON may contain forward slashes.
**How to avoid:** Normalize all paths at the API boundary. Use `path.resolve()` for server operations. Send normalized paths (forward slashes) to the client. The existing watcher.ts already does `filename.replace(/\\/g, "/")`.
**Warning signs:** 404 errors on directory listings, "ENOENT" errors with mixed separator paths.

### Pitfall 2: Permission Errors on Restricted Directories
**What goes wrong:** `readdir` throws EACCES/EPERM on system directories the user can't read.
**Why it happens:** Users may navigate to `/`, `C:\Windows`, or other restricted paths.
**How to avoid:** Wrap readdir in try/catch, return an error response with a user-friendly message. Filter out known system directories in the UI.
**Warning signs:** Unhandled promise rejections in the Bun server.

### Pitfall 3: Race Condition on Project Switch
**What goes wrong:** User switches projects while Claude is processing, watcher events from old directory get mixed with new state.
**Why it happens:** The pipeline switch involves multiple async steps (stop watcher, rebuild state, restart).
**How to avoid:** Lock project switching while `isProcessing` is true. Show a warning if Claude is active. Sequence the switch: kill Claude first, then stop watcher, then rebuild state.
**Warning signs:** State panel shows data from the wrong project.

### Pitfall 4: Symlink Loops in Directory Traversal
**What goes wrong:** Following symlinks can cause infinite recursion or very long response times.
**Why it happens:** Some directories contain symlinks pointing to parent directories.
**How to avoid:** Don't follow symlinks in the directory listing. Use `lstat` instead of `stat` if needed, or just check `dirent.isSymbolicLink()` and skip or mark them.
**Warning signs:** API requests that never complete or return enormous results.

### Pitfall 5: Large Directory Listings
**What goes wrong:** Directories like `node_modules` contain thousands of entries, causing slow responses and UI jank.
**Why it happens:** No pagination or filtering on the API.
**How to avoid:** Limit responses (e.g., first 200 entries), filter hidden files by default, skip `node_modules` in the tree view (or collapse it with a count badge).
**Warning signs:** Multi-second API responses, browser memory spikes.

### Directory Listing with GSD Project Detection
```typescript
// server/fs-api.ts
import { readdir, access } from "node:fs/promises";
import { join, resolve, normalize } from "node:path";
import { homedir } from "node:os";

export interface FileSystemEntry {
  name: string;
  path: string;       // normalized, forward-slash
  isDirectory: boolean;
  isGsdProject: boolean;  // has .planning/ subdirectory
}

const HIDDEN_DIRS = new Set(["node_modules", ".git", "$RECYCLE.BIN", "System Volume Information"]);

export async function listDirectory(dirPath: string): Promise<FileSystemEntry[]> {
  const resolved = resolve(dirPath);
  const entries = await readdir(resolved, { withFileTypes: true });

  const results: FileSystemEntry[] = [];
  for (const entry of entries) {
    // Skip hidden files (except .planning), skip known noise dirs
    if (entry.name.startsWith(".") && entry.name !== ".planning") continue;
    if (HIDDEN_DIRS.has(entry.name)) continue;

    const fullPath = join(resolved, entry.name).replace(/\\/g, "/");
    let isGsdProject = false;

    if (entry.isDirectory()) {
      try {
        await access(join(resolved, entry.name, ".planning"));
        isGsdProject = true;
      } catch { /* not a GSD project */ }
    }

    results.push({
      name: entry.name,
      path: fullPath,
      isDirectory: entry.isDirectory(),
      isGsdProject,
    });
  }

  // Sort: directories first, then alphabetically
  results.sort((a, b) => {
    if (a.isDirectory !== b.isDirectory) return a.isDirectory ? -1 : 1;
    return a.name.localeCompare(b.name);
  });

  return results;
}
```

### Recent Projects Persistence
```typescript
// server/recent-projects.ts
import { readFile, writeFile, mkdir } from "node:fs/promises";
import { join } from "node:path";
import { homedir } from "node:os";

const RECENT_FILE = join(homedir(), ".gsd", "recent-projects.json");

export interface RecentProject {
  path: string;
  name: string;
  lastOpened: number; // timestamp
  isGsdProject: boolean;
}

export async function getRecentProjects(): Promise<RecentProject[]> {
  try {
    const data = await readFile(RECENT_FILE, "utf-8");
    return JSON.parse(data);
  } catch {
    return [];
  }
}

export async function addRecentProject(project: RecentProject): Promise<void> {
  const projects = await getRecentProjects();
  const filtered = projects.filter(p => p.path !== project.path);
  filtered.unshift({ ...project, lastOpened: Date.now() });
  const trimmed = filtered.slice(0, 20); // Keep last 20
  await mkdir(join(homedir(), ".gsd"), { recursive: true });
  await writeFile(RECENT_FILE, JSON.stringify(trimmed, null, 2));
}
```

### useProjectBrowser Hook
```typescript
// hooks/useProjectBrowser.ts
import { useState, useCallback } from "react";
import type { FileSystemEntry } from "../server/fs-api";

export function useProjectBrowser() {
  const [currentPath, setCurrentPath] = useState<string | null>(null);
  const [entries, setEntries] = useState<FileSystemEntry[]>([]);
  const [loading, setLoading] = useState(false);
  const [error, setError] = useState<string | null>(null);

  const browse = useCallback(async (path: string) => {
    setLoading(true);
    setError(null);
    try {
      const res = await fetch(`/api/fs/list?path=${encodeURIComponent(path)}`);
      if (!res.ok) throw new Error(await res.text());
      const data = await res.json();
      setEntries(data);
      setCurrentPath(path);
    } catch (err) {
      setError(err instanceof Error ? err.message : String(err));
    } finally {
      setLoading(false);
    }
  }, []);

  const openProject = useCallback(async (path: string) => {
    const res = await fetch("/api/project/switch", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ path }),
    });
    if (!res.ok) throw new Error(await res.text());
    return res.json();
  }, []);

  return { currentPath, entries, loading, error, browse, openProject };
}
```

## Open Questions

1. **Starting path for the file browser**
   - What we know: User's home directory (`os.homedir()`) is the most natural default
   - What's unclear: Should it default to the current working directory of the server instead?
   - Recommendation: Default to home directory, but also show the current project path prominently. Add breadcrumb navigation for quick path switching.

2. **New project initialization**
   - What we know: Phase 7 covers SESS-01 (new project onboarding with `/gsd:new-project`)
   - What's unclear: Should 06.1 handle `mkdir` + initialize `.planning/` or just navigate to existing directories?
   - Recommendation: 06.1 should support creating new empty directories. Actual GSD project initialization (creating `.planning/` files) should remain in Phase 7's session flow. 06.1 provides the "open folder" UX.

3. **Multi-project support scope**
   - What we know: SESS-03 requires "project selector for multiple GSD projects sorted by most recent activity" in Phase 7
   - What's unclear: How much of multi-project support belongs in 06.1 vs Phase 7
   - Recommendation: 06.1 builds the infrastructure (recent projects file, project switching API, pipeline hot-switch). Phase 7 builds the polished UI (project selector, sorting, onboarding flow).
