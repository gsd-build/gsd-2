    - getSettings returns empty objects when config files don't exist (no crash)
    - saveSettings("global", data) writes to ~/.gsd/defaults.json, creating ~/.gsd/ dir if needed
    - saveSettings("project", data, planningDir) writes to planningDir/config.json
    - saveSettings merges with existing data (partial update, not overwrite)
    - GET /api/settings returns merged settings with tier indicators
    - PUT /api/settings with { tier, settings } saves to correct tier
    - PUT /api/settings with tier "project" but no planningDir returns 400
  </behavior>
  <action>
    1. Create `settings-api.ts` following route dispatcher pattern:
       - `getSettings(planningDir: string): Promise<{ global: Record<string, unknown>; project: Record<string, unknown>; merged: Record<string, unknown> }>` — reads `~/.gsd/defaults.json` and `planningDir/config.json`, returns all three objects. Uses `readFile` with try/catch returning `{}` on missing file.
       - `saveSettings(tier: "global" | "project", settings: Record<string, unknown>, planningDir?: string): Promise<void>` — reads existing file, spreads new settings over old, writes back. Creates parent dirs with `mkdir({ recursive: true })`.
       - `handleSettingsRequest(req: Request, url: URL, planningDir: string): Promise<Response | null>`:
         - `GET /api/settings` — calls getSettings, returns JSON
         - `PUT /api/settings` — parses body `{ tier, settings }`, calls saveSettings, returns 200
         - Returns null for unmatched routes
       - Use `os.homedir()` for global path, `path.join` for all paths

    2. Write tests in `tests/settings-api.test.ts`:

       - Test all behaviors above
       - Mock `os.homedir()` to point to temp dir for global settings isolation

  <behavior>
    - POST /api/assets/upload accepts multipart form data with file field
    - Upload stores file in planningDir/assets/ preserving original filename
    - Upload creates assets/ directory if it doesn't exist
    - Upload rejects files over 50MB with 413 status
    - Upload rejects unsupported file types with 400 status
    - Supported types: png, jpg, jpeg, gif, svg, mp4, webm, pdf, md, txt
    - GET /api/assets/list returns array of { name, path, type, size, createdAt }
    - DELETE /api/assets/:name removes the file from assets/
    - DELETE returns 404 if file doesn't exist
  </behavior>
  <action>
    1. Create `assets-api.ts` following route dispatcher pattern:
       - Define `SUPPORTED_EXTENSIONS` set: `png, jpg, jpeg, gif, svg, mp4, webm, pdf, md, txt`
       - Define `MAX_FILE_SIZE = 50 * 1024 * 1024` (50MB)
       - `handleAssetsRequest(req: Request, url: URL, planningDir: string): Promise<Response | null>`:
         - `POST /api/assets/upload`: parse formData, validate extension against SUPPORTED_EXTENSIONS, validate size against MAX_FILE_SIZE, write to `planningDir/assets/<filename>` using `Bun.write()`, return `{ name, path, type, size }`
         - `GET /api/assets/list`: scan `planningDir/assets/` directory, return array of asset metadata with `stat()` for size and ctime
         - `DELETE /api/assets/:name`: extract filename from pathname, resolve path within `planningDir/assets/`, delete file, return 200. Path traversal check: reject if resolved path is not under assets dir.
         - Handle duplicate filenames on upload: append `-1`, `-2` suffix before extension
       - Export `SUPPORTED_EXTENSIONS` and `MAX_FILE_SIZE` for testing

    2. Wire into `server.ts`:
       - Import `handleAssetsRequest` and `handleSettingsRequest`
       - Add route blocks for `/api/settings/` and `/api/assets/` following existing pattern
       - Pass `planningDir` (derived from pipeline) to both handlers

    3. Write tests in `tests/assets-api.test.ts`:
       - Use temp directory as planningDir
       - Test upload with valid file (create small test buffer)
       - Test rejection of oversized file (check 413 response)
       - Test rejection of unsupported type (.exe)
       - Test listing returns uploaded files
       - Test delete removes file, 404 on missing
  </action>
