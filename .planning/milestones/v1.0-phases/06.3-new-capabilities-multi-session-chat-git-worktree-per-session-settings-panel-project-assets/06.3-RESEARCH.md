# Phase 06.3: New Capabilities -- Multi-session Chat, Git Worktree, Settings, Assets - Research

**Researched:** 2026-03-11
**Domain:** Multi-session process management, git worktree automation, settings UI, asset management
**Confidence:** HIGH

## Summary

This phase adds five interconnected capabilities to Mission Control: multi-session chat (up to 4 parallel Claude processes), git worktree isolation per session, a settings panel, project asset storage with gallery view, and split milestone visualization. The existing architecture is well-suited for extension -- `ClaudeProcessManager` is already a clean class that can be instantiated multiple times, the WebSocket topic-based pub/sub pattern naturally extends to session-scoped topics, and the `ViewType` discriminated union cleanly supports new view kinds.

The primary complexity lies in the multi-session layer: the `pipeline.ts` currently assumes a single `ClaudeProcessManager` and a single `activeClient`. This must become a session registry (Map of session ID to manager+client pairs). The WebSocket protocol needs session IDs on all chat messages to route events to the correct client tab. Git worktree operations are straightforward shell commands but need careful error handling for Windows paths and cleanup on session close.

**Primary recommendation:** Build the session registry in pipeline.ts first (Map-based session management), then layer worktree isolation, settings, and assets on top as independent feature slices.

### Pattern 1: Session Registry (server-side)
**What:** A Map-based registry that tracks all active sessions per project. Each session has its own `ClaudeProcessManager`, message state, and optional worktree path.
**When to use:** All multi-session operations route through this registry.
**Example:**
```typescript
// session-manager.ts
interface SessionState {
  id: string;
  name: string;            // "Chat 1" or semantic name
  slug: string;            // kebab-case for filesystem
  processManager: ClaudeProcessManager;
  activeClient: ServerWebSocket | null;
  worktreePath: string | null;   // null if worktree isolation off
  worktreeBranch: string | null;
  createdAt: number;
}

class SessionManager {
  private sessions = new Map<string, SessionState>();
  private maxSessions = 4;

  createSession(cwd: string, opts?: { forkFromSessionId?: string }): SessionState { ... }
  getSession(id: string): SessionState | undefined { ... }
  closeSession(id: string): Promise<void> { ... }
  renameSession(id: string, newName: string): Promise<void> { ... }
  listSessions(): SessionState[] { ... }
}
```

### Pattern 2: Session-scoped WebSocket Protocol
**What:** All chat messages include a `sessionId` field so the server routes events to the correct client tab.
**When to use:** Every chat request/response in the WebSocket protocol.
**Example:**

// Extended ChatResponse

// ws-server message handler
if (parsed.type === "chat" && parsed.sessionId) {
  const session = sessionManager.getSession(parsed.sessionId);

  session.activeClient = ws;

### Anti-Patterns to Avoid
- **Singleton ProcessManager assumption:** The pipeline currently uses a single `processManager` variable. Do NOT try to add session support by adding arrays or parallel variables -- use a proper SessionManager class.
- **Session state in useChat hook:** Do NOT store per-session state in a single `useChat` call. Create a `useSessionManager` that manages multiple `useChat`-like states indexed by session ID.
- **Blocking git operations:** All git worktree commands must be async (Promise-wrapped spawn). Never use `execSync` or `spawnSync` -- they block the event loop.
- **Worktree paths with spaces:** Always quote paths passed to git commands. Use array args (not shell: true) to avoid escaping issues.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| npm registry search | Custom scraper | `fetch("https://registry.npmjs.org/-/v1/search?text=...")` | Official REST API, free, no auth needed |
| Unique IDs | Counter-based IDs | `crypto.randomUUID()` | Collision-free, built-in |
| File type detection | Extension parsing | Lookup table by extension | Known set of supported types |
| Slug generation | Custom regex | Simple kebab-case: `name.toLowerCase().replace(/[^a-z0-9]+/g, '-').replace(/^-|-$/g, '')` | Predictable, filesystem-safe |
| Thumbnail generation | Server-side image processing | Client-side `<img>` with CSS `object-fit: cover` | Browser handles resizing natively |
| .gitignore editing | Manual file parsing | `fs.readFile` + check if `.worktrees/` line exists + append if not | Simple string operation |

**Key insight:** This phase has no image processing or complex media handling. Thumbnails are just `<img>` tags with constrained CSS dimensions. Video thumbnails can use `<video>` tag with `poster` attribute or first frame. PDFs get a file icon.

## Common Pitfalls

### Pitfall 1: Race Condition on Session Close
**What goes wrong:** User closes a session tab while Claude is still processing. The process kill races with the event stream, causing orphaned events routed to a dead session.
**Why it happens:** ClaudeProcessManager.kill() is async, and events may be in-flight on the NDJSON parser buffer.
**How to avoid:** On session close: (1) set `session.activeClient = null` immediately to stop routing, (2) then call `processManager.kill()`, (3) then remove from registry. Events arriving after step 1 are silently dropped.
**Warning signs:** `sendToClient` errors in server logs after tab close.

### Pitfall 2: Worktree Path on Windows
**What goes wrong:** Git worktree paths use backslashes on Windows, but git commands internally expect forward slashes in some contexts.
**Why it happens:** `path.resolve()` returns backslash paths on Windows, git is inconsistent about path handling.
**How to avoid:** Normalize all paths to forward slashes before passing to git commands, consistent with existing `replace(/\\/g, "/")` pattern throughout the codebase.
**Warning signs:** "fatal: not a valid directory" errors from git worktree commands.

### Pitfall 3: Worktree Branch Name Collisions
**What goes wrong:** Two sessions with similar names create the same branch `session/my-feature`, second creation fails.
**Why it happens:** Slug generation from semantic names can collide.
**How to avoid:** Append session ID suffix to branch names: `session/<slug>-<short-id>`. Or check branch existence before creating and append numeric suffix.
**Warning signs:** `git worktree add` returns error about existing branch.

### Pitfall 4: Session Rename with Active Worktree
**What goes wrong:** User renames a session tab while worktree is active. The worktree directory and branch need to be renamed atomically, but there's no atomic rename for worktrees.
**Why it happens:** Git has no `git worktree rename` command. Renaming requires remove + add.
**How to avoid:** Worktree rename = create new worktree + copy changes + remove old. Simpler approach: disallow rename when worktree isolation is ON (show tooltip explaining why). Or keep worktree path fixed at creation time and only rename the display label.
**Warning signs:** Orphaned worktree directories after rename failures.

### Pitfall 5: Multiple WebSocket Connections
**What goes wrong:** useChat creates its own WebSocket connection. With multi-session, creating 4 WebSocket connections wastes resources.
**Why it happens:** Current useChat hook creates a standalone WebSocket.
**How to avoid:** Share a single WebSocket connection. The session ID field on messages handles routing. The `useSessionManager` hook manages one WebSocket and demuxes messages to per-session state by `sessionId`.
**Warning signs:** 4+ WebSocket connections visible in browser dev tools.

### Pitfall 6: Settings Panel Config Merge
**What goes wrong:** Saving a setting to project tier doesn't clear the global default, leading to confusion about which value is active.
**Why it happens:** Naive merge doesn't track provenance.
**How to avoid:** The settings API returns `{ global, project, merged }` and the UI shows "(project override)" badge next to any key that exists in project config. A "Reset to default" button removes the key from project config.
**Warning signs:** Settings not taking effect after save.

### Pitfall 7: Asset Upload Size
**What goes wrong:** User drags a 500MB video into chat, server buffers entire file in memory.
**Why it happens:** `req.formData()` loads entire body into memory.
**How to avoid:** Set a reasonable limit (50MB). Bun.serve supports `maxRequestBodySize` option. Reject files over limit with clear error message before buffering.
**Warning signs:** Server OOM on large file upload.

### Git Worktree Create (server-side)
```typescript
// Follows git-api.ts spawn pattern
function spawnGit(cwd: string, args: string[]): Promise<{ ok: boolean; stdout: string; stderr: string }> {
  return new Promise((resolve) => {
    const proc = nodeSpawn("git", args, {
      cwd,
      stdio: ["ignore", "pipe", "pipe"],
      shell: false,
    });
    let stdout = "", stderr = "";
    proc.stdout!.on("data", (c: Buffer) => { stdout += c.toString(); });
    proc.stderr!.on("data", (c: Buffer) => { stderr += c.toString(); });
    proc.on("close", (code) => {
      resolve({ ok: code === 0, stdout: stdout.trim(), stderr: stderr.trim() });
    });
    proc.on("error", (err) => {
      resolve({ ok: false, stdout: "", stderr: err.message });
    });
  });
}

// Create worktree for a session
async function createSessionWorktree(
  repoRoot: string,
  sessionSlug: string,
): Promise<{ worktreePath: string; branchName: string } | { error: string }> {
  const worktreePath = join(repoRoot, ".worktrees", sessionSlug).replace(/\\/g, "/");
  const branchName = `session/${sessionSlug}`;

  // Ensure .worktrees/ is in .gitignore
  await ensureGitignoreEntry(repoRoot, ".worktrees/");

  const result = await spawnGit(repoRoot, [
    "worktree", "add", "-b", branchName, worktreePath,
  ]);

  if (!result.ok) {
    return { error: result.stderr || "Failed to create worktree" };
  }
  return { worktreePath, branchName };
}
```

### npm Registry Search
```typescript
// For marketplace -- simple fetch, no library needed
interface NpmPackage {
  name: string;
  description: string;
  version: string;
  date: string;
  keywords: string[];
}

async function searchNpmPackages(query: string, limit = 20): Promise<NpmPackage[]> {
  const url = `https://registry.npmjs.org/-/v1/search?text=${encodeURIComponent(query)}&size=${limit}`;
  const res = await fetch(url);
  if (!res.ok) return [];
  const data = await res.json();
  return (data.objects || []).map((obj: any) => ({
    name: obj.package.name,
    description: obj.package.description || "",
    version: obj.package.version,
    date: obj.package.date,
    keywords: obj.package.keywords || [],
  }));
}
```

## Open Questions

1. **Semantic name derivation algorithm**
   - What we know: Should work "like Claude Desktop" -- derive from first user message + response
   - What's unclear: Exact algorithm. Could use first 3-5 words of user message, or ask Claude to summarize, or parse response topic.
   - Recommendation: Use first user message, truncated to ~30 chars, cleaned to title case. Simple and predictable. No AI call needed for naming.

2. **Session metadata persistence location**
   - What we know: Persist IDs, names, tab order -- not full history
   - What's unclear: Where exactly? `.planning/.mission-control-session.json` already exists for layout prefs
   - Recommendation: Extend the existing session file with a `sessions` array. Consistent with SERV-07 pattern.

3. **Worktree rename mechanics**
   - What we know: Git has no `git worktree rename`. User wants tab+dir+branch to match.
   - What's unclear: Is rename worth the complexity (remove+recreate risks losing uncommitted work)?
   - Recommendation: Keep worktree path/branch fixed at creation, only rename display label. Note in settings tooltip. This is the safe approach.

4. **Marketplace install mechanism**
   - What we know: "One-click install (runs npx/npm in background, prompts restart)"
   - What's unclear: What exactly gets installed? npm packages into project? Into global ~/.gsd/plugins/?
   - Recommendation: Install to `~/.gsd/plugins/<package-name>/` via `npm install --prefix ~/.gsd/plugins/<name> <package>`. Scan plugins dir on startup for skills/commands.
