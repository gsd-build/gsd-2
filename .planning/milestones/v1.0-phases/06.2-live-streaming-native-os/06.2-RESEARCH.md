# Phase 6.2: Live Streaming Architecture + Native OS Integration - Research

**Researched:** 2026-03-10
**Domain:** WebSocket streaming, Claude Code CLI integration, native OS dialogs, sidebar/layout restructure
**Confidence:** HIGH

## Summary

This phase transforms Mission Control from a semi-static dashboard into a live streaming IDE-like command center. The core work divides into four domains: (1) enhancing the Claude Code process manager to stream all event types including sub-agent events with proper tagging, (2) replacing the in-panel file browser with a native OS folder dialog via server-side process spawning, (3) restructuring the sidebar into a VS Code-style project tree with expandable sections, and (4) converting the right panel from tab-based multi-panel layout to a single-column view driven by sidebar selection.

The existing codebase provides strong foundations: `ClaudeProcessManager` already spawns Claude with NDJSON streaming, `createNdjsonParser` handles stream parsing, and the WebSocket pub/sub pattern (`planning-state` and `chat` topics) is established. The `switchProject()` function in pipeline.ts already handles watcher close, process kill, state rebuild, and broadcast -- it needs enhancement for full client-side state reload signals.

**Primary recommendation:** Extend the existing NDJSON pipeline to surface all Claude Code stream-json event types (including sub-agent `parent_tool_use_id` tagging), add `--dangerously-skip-permissions` as a default flag with settings toggle, and implement server-side native OS dialog spawning behind an abstracted API endpoint.

### Pattern 1: Sub-Agent Event Tagging
**What:** Claude Code stream-json events from sub-agents include a `parent_tool_use_id` field that links them to the parent tool call that spawned them.
**When to use:** Whenever processing streaming events to distinguish main-process vs sub-agent output.
**Example:**
```typescript
// Source: https://platform.claude.com/docs/en/agent-sdk/streaming-output
// StreamEvent structure from Claude Code CLI stream-json:
interface StreamEventEnhanced {
  type: "stream_event" | "system" | "assistant" | "result";
  event?: {
    type: "message_start" | "content_block_start" | "content_block_delta" | "content_block_stop" | "message_delta" | "message_stop";
    index?: number;
    delta?: { type: string; text?: string; partial_json?: string };
    content_block?: { type: string; name?: string; id?: string };
  };
  parent_tool_use_id?: string | null; // Non-null = sub-agent event
  session_id?: string;
  uuid?: string;
  result?: string;
  subtype?: string;
}
```

### Pattern 2: Native OS Dialog via Server-Side Spawn
**What:** Server endpoint spawns a platform-specific process to show a native folder dialog, captures stdout for the selected path.
**When to use:** When "Open Folder" is triggered from the dashboard.
**Example:**
```typescript
// Windows: PowerShell FolderBrowserDialog
import { spawn as nodeSpawn } from "node:child_process";

async function openFolderDialog(): Promise<string | null> {
  const isWindows = process.platform === "win32";

  if (isWindows) {
    const ps = nodeSpawn("powershell", [
      "-NoProfile", "-Command",
      `Add-Type -AssemblyName System.Windows.Forms;
       $dlg = New-Object System.Windows.Forms.FolderBrowserDialog;
       $dlg.Description = 'Select Project Folder';
       $dlg.ShowNewFolderButton = $true;
       if ($dlg.ShowDialog() -eq 'OK') { $dlg.SelectedPath } else { '' }`
    ], { stdio: ["ignore", "pipe", "pipe"] });

    return new Promise((resolve) => {
      let stdout = "";
      ps.stdout!.on("data", (chunk: Buffer) => { stdout += chunk.toString(); });
      ps.on("close", (code) => {
        const path = stdout.trim();
        resolve(path || null);
      });
    });
  }

  // Linux: zenity
  const zen = nodeSpawn("zenity", [
    "--file-selection", "--directory", "--title=Select Project Folder"
  ], { stdio: ["ignore", "pipe", "pipe"] });

  return new Promise((resolve) => {
    let stdout = "";
    zen.stdout!.on("data", (chunk: Buffer) => { stdout += chunk.toString(); });
    zen.on("close", (code) => {
      resolve(code === 0 ? stdout.trim() : null);
    });
  });
}
```

### Pattern 4: Permission Prompt via WebSocket
**What:** When Claude Code hits a permission prompt (only when --dangerously-skip-permissions is DISABLED), the server forwards it to the client as a WebSocket event, client shows a blocking modal, user response is piped back.
**When to use:** Permission mode enabled in settings.
**Example:**
```typescript
// Server: detect permission prompt in stream and forward
// The --permission-prompt-tool flag lets Claude call an MCP tool for permission decisions
// Simpler approach for v1: detect permission prompt patterns in stderr/stdout

// WebSocket protocol extension:
interface PermissionPromptEvent {
  type: "permission_prompt";
  toolName: string;
  toolInput: string;
  promptId: string;  // Unique ID for correlating response
}

interface PermissionResponse {
  type: "permission_response";
  promptId: string;
  action: "approve" | "always_allow" | "deny";
}
```

### Anti-Patterns to Avoid
- **Multiple WebSocket connections per feature:** Keep using the single WS connection with topic-based pub/sub. Do NOT create separate connections for activity, permissions, etc.
- **Polling for permission responses:** Use the existing WebSocket bidirectional channel. Client sends permission_response back on the same connection.
- **Spawning persistent dialog process:** Dialog processes should be one-shot (spawn, capture stdout, exit). Do NOT keep a dialog process alive.
- **Storing stream events on server:** Events are fire-and-forget to connected WebSocket clients. Server-side buffering wastes memory. If no client is connected, events are dropped.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Native folder dialog | Custom Electron window | PowerShell/zenity spawn | Abstraction-ready for future Tauri migration; zero dependencies |
| Git log parsing | Custom git output parser | `git log --format=json` or simple `git log --oneline` spawn | Git output is well-structured; parsing custom formats is fragile |
| Stream event buffering | Custom circular buffer | Direct WebSocket publish | Events are transient; clients that miss them get full state on reconnect |
| Permission modal state | Custom modal manager | Simple React state + portal | Single modal at a time; no modal stacking needed |

**Key insight:** The existing NDJSON parser and WebSocket pub/sub infrastructure handle 80% of the streaming work. The main additions are protocol extensions (new event types) and UI restructuring, not new infrastructure.

## Common Pitfalls

### Pitfall 1: PowerShell Dialog Blocking the Server
**What goes wrong:** PowerShell's FolderBrowserDialog runs synchronously and blocks the spawning thread until the user closes it.
**Why it happens:** PowerShell dialog is GUI-modal. If spawned incorrectly, it could block the Bun event loop.
**How to avoid:** Node's `child_process.spawn` with piped stdio already runs in a separate process. The server endpoint should return immediately with a "dialog opened" response, then use a follow-up mechanism (WebSocket event or polling endpoint) to deliver the selected path.
**Warning signs:** Server stops responding to other requests while dialog is open.

### Pitfall 2: Sub-Agent Event Interleaving
**What goes wrong:** Main process and sub-agent events arrive interleaved on the same stdout stream, making it impossible to distinguish which is which.
**Why it happens:** Claude Code CLI multiplexes all events (main + sub-agents) onto the same NDJSON output stream.
**How to avoid:** Use the `parent_tool_use_id` field present on StreamEvent. Non-null value means sub-agent event. The existing NDJSON parser already handles the raw parsing; just propagate the field through ChatResponse.
**Warning signs:** Chat shows sub-agent tool calls as if they're main process actions.

### Pitfall 3: WebSocket Reconnect Drops State on Project Switch
**What goes wrong:** When project switches, the WebSocket connection may drop and reconnect, but the client still shows stale state from the previous project.
**Why it happens:** The existing reconnect logic sends "refresh" on reconnect, but chat messages and activity feed are client-side state that isn't cleared.
**How to avoid:** On receiving a "project_switched" WebSocket event, client must: (1) clear chat messages, (2) clear activity feed, (3) reset sidebar selection to default view, (4) wait for new "full" state to arrive.
**Warning signs:** Previous project's chat messages visible after switching.

### Pitfall 4: Permission Prompt Deadlock
**What goes wrong:** Claude Code process blocks waiting for permission approval, but the event that triggered the permission prompt hasn't been forwarded to the client yet.
**Why it happens:** If the NDJSON parser doesn't recognize permission prompt events, or if the WebSocket message gets dropped due to backpressure.
**How to avoid:** For v1 with --dangerously-skip-permissions as default, permission prompts only appear when explicitly disabled in settings. This is a secondary code path. When implementing, use the `--permission-prompt-tool` MCP approach which provides a structured callback mechanism rather than stdin pipe.
**Warning signs:** Claude Code process hangs indefinitely with no visible output.

### Pitfall 5: ProjectBrowser Removal Breaking Imports
**What goes wrong:** Removing ProjectBrowser.tsx breaks imports in Sidebar.tsx, useProjectBrowser.ts, and FileTreeNode.tsx.
**Why it happens:** Multiple components reference ProjectBrowser and its types.
**How to avoid:** Remove ProjectBrowser.tsx, FileTreeNode.tsx, and useProjectBrowser.ts together. Update Sidebar.tsx to use native dialog trigger instead. Keep the project switch logic (it's reused by the native dialog flow).
**Warning signs:** TypeScript compilation errors after removal.

// In dialog-api.ts:
export async function handleDialogRequest(
  req: Request,
  url: URL
): Promise<Response | null> {
  if (url.pathname === "/api/dialog/open-folder" && req.method === "POST") {
    const selectedPath = await openFolderDialog();
    if (selectedPath) {
      return Response.json({ path: selectedPath.replace(/\\/g, "/") });
    }
    return Response.json({ cancelled: true });
  }
  return null;
}

## Open Questions

1. **Permission prompt detection in stream-json**
   - What we know: Claude Code has `--permission-prompt-tool` for MCP-based permission handling. Stream-json output does not explicitly document a "permission_prompt" event type.
   - What's unclear: Exact mechanism for detecting and responding to permission prompts when running Claude Code in `-p` (print/SDK) mode without `--dangerously-skip-permissions`.
   - Recommendation: For v1, default to `--dangerously-skip-permissions`. Implement permission UI as a secondary feature. When implementing, use `--permission-prompt-tool` with a local MCP server that communicates via the existing WebSocket channel.

2. **GSD checkpoint detection for History view**
   - What we know: User wants History to show only GSD checkpoint snapshots, not every git commit.
   - What's unclear: How GSD checkpoints are identified in git history (commit message prefix? tag? specific file changes?).
   - Recommendation: Use a convention-based approach: GSD checkpoint commits have a recognizable prefix (e.g., `gsd:checkpoint` or `docs(XX):`) that can be filtered with `git log --grep`.

3. **"New Window" implementation**
   - What we know: User wants "New Window" to open a new browser tab with its own independent Claude Code session.
   - What's unclear: Whether the new tab connects to the same server (shared state) or needs a separate server instance.
   - Recommendation: New tab opens same URL (localhost:4000). Each tab gets its own WebSocket connection. Claude Code process is per-project (shared), so multiple tabs see the same stream. True independent sessions require multi-chat (deferred).
