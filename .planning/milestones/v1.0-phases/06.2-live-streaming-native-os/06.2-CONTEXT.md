# Phase 6.2: Live Streaming Architecture + Native OS Integration - Context

**Gathered:** 2026-03-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Stream Claude Code process output (including sub-agents) to dashboard in real-time. Replace in-panel file browser with native OS file picker. Restructure sidebar navigation and panel layout so clicking a tree item shows a single-column view instead of the current 4-panel constant layout. Dashboard state fully refreshes on project switch.

</domain>

<decisions>
## Implementation Decisions

### Permission Prompt UX
- Blocking modal overlay when Claude Code hits a permission prompt
- Three-button modal: Approve / Always Allow (whitelist tool type for session) / Deny
- Permission modals only appear when `--dangerously-skip-permissions` flag is DISABLED
- Default launch uses `--dangerously-skip-permissions` — no modals in default mode
- Claude's Discretion: mechanism for piping approve/deny back to Claude CLI process stdin

### Native OS File Dialog
- "Open Folder" triggers native OS file dialog via server-side spawn (PowerShell on Windows, zenity on Linux)
- Abstracted behind API endpoint (`POST /api/dialog/open-folder`) for future Tauri migration
- Replaces the in-panel ProjectBrowser tree component entirely — no fallback browser
- When Tauri wraps the app later, swap server-side dialog for Tauri's `dialog.open()` API — same frontend code

### Sidebar Restructure
- Sidebar contains only: New Window button, Open Folder button, and Project list
- Remove Activity, Verify, History as top-level nav items (NavItems.tsx)
- Each project in the list expands to a sub-tree showing:
  - **History**: GSD checkpoint snapshots only (not every git commit) — each with "Revert to this" action
  - **Milestones**: Milestone entries from `.planning/`
  - **Slices (Phases)**: Phase entries with status indicators
  - **Chat**: Single chat session (multi-chat deferred)
- "New Window" opens a new browser tab with its own independent Claude Code session

### Panel Layout Change
- Clicking a sidebar tree item (milestone, slice, chat) opens a SINGLE column view in the right panel area
- Replaces the current 4-column constant layout (Milestone View + Slice Detail + Active Task + Chat all visible)
- One view at a time, full width of the right area
- Click milestone → see milestone detail only. Click slice → see slice detail only. Click chat → see chat only.

### Streaming Architecture
- Claude Code process streams all events (tool_use, text, thinking, result) via WebSocket — enhance existing NDJSON pipeline
- Sub-agent/Task tool spawned processes also stream their events (not just main process)
- `--dangerously-skip-permissions` flag on by default with option to disable in settings
- After project switch, entire dashboard state reloads (WebSocket reconnects, panel refreshes)

### Claude's Discretion
- Event stream filtering logic (which events to surface, which to suppress)
- WebSocket message protocol extensions for permission prompts
- How sub-agent events are distinguished from main process events in the stream
- Technical approach for state reload on project switch (existing `switchProject()` in pipeline.ts)

</decisions>

<specifics>
## Specific Ideas

- User envisions sidebar like VS Code's Explorer — project tree with expandable sections
- History shows only GSD checkpoint commits (meaningful save points before plan execution), not every git commit
- Future Tauri desktop wrapper planned — all native OS integration should be abstraction-ready
- User wants the right panel to feel like switching terminal tabs — one view at a time, not a crowded multi-column layout

</specifics>

<code_context>
## Existing Code Insights

### Reusable Assets
- `ClaudeProcessManager` (claude-process.ts): Already spawns Claude with NDJSON streaming, per-message spawn with --resume. Needs `--dangerously-skip-permissions` flag added.
- `createNdjsonParser` (ndjson-parser.ts): Callback-based NDJSON parser already handles stream events. Can be extended for permission events.
- `StreamEvent` type (chat-types.ts): Covers system/assistant/result/stream_event. Needs extension for permission_prompt type.
- `NavItems.tsx`: Has Activity/Verify/History as stubs — will be replaced by project sub-tree.
- `ProjectBrowser.tsx`: In-panel file tree browser — will be removed, replaced by native OS dialog.
- `pipeline.ts switchProject()`: Already handles project switching with watcher close, process kill, state rebuild, broadcast. Needs enhancement for full state reload.

### Established Patterns
- Node's `child_process.spawn` (not Bun.spawn) for Windows reliability
- CLAUDECODE env var stripping to avoid nested session rejection
- Topic-based pub/sub on WebSocket (`planning-state` and `chat` topics)
- `publishChat()` for broadcasting chat events to all subscribed clients
- Route dispatcher pattern for API endpoints (fs-api.ts)

### Integration Points
- `Sidebar.tsx`: Entry point for sidebar restructure — currently renders NavItems + ProjectList + ProjectBrowser
- `ws-server.ts`: WebSocket server needs new message types for permission prompts
- `pipeline.ts wireProcessManagerEvents()`: Event forwarding logic — needs sub-agent awareness
- `useChat.ts`: Client-side event processing — needs permission prompt handling
- `AppShell` or equivalent: Needs to manage single-column view switching based on sidebar selection

</code_context>

<deferred>
## Deferred Ideas

- **Multi-chat sessions per project**: User wants up to 4 chat tabs per project, each with its own Claude Code subprocess (simulating multi-terminal). Requires session management architecture — future phase.
- **Git worktree per session**: Each chat session could create its own git worktree for parallel implementations on the same codebase — future phase.
- **Claude Code settings panel**: Settings view in the right panel for managing permissions flag, custom slash commands, skills, custom sub-agents (non-GSD). New capability — future phase.
- **Revert to checkpoint**: History shows GSD checkpoints with "Revert to this" action — UI exists in this phase, but the actual git revert mechanism may need its own implementation work.

</deferred>
