| K001 | M005-8pv12q planning | Evidence discipline design | Training data recall is never "observed" — it is a source of concepts and hypotheses, not facts. Version numbers, API signatures, CLI flags, config schemas from training data require verification before acting on them. | User constraint. The `actions/checkout@v4` example: agent states it with confidence, it's wrong (current is v6). Every recalled fact needs a verification path. |
| K002 | M005-8pv12q planning | Pipeline design | Structure Over Instruction — telling an agent to do something is behavioral and ignorable. Structuring the pipeline so it must do the thing is architectural and reliable. ARL Principle 1. | Three iterations of M005-8pv12q design proved this: standalone templates (behavioral, ignored) → constraint labels (labeled the problem) → unknowns inventory + fact-checker (structural). |
| K003 | M005-8pv12q planning | Pipeline design | AI cannot reliably self-evaluate (ARL Principle 3). An agent classifying which of its own claims came from training data shares the same blind spots that produced them. Independent verification (fact-checker on haiku) is the structural solution. | M005-8pv12q prompts do self-classification as a useful interim. M005 adds the fact-check service layer for structural enforcement. |
| K004 | M005-8pv12q planning | Fact-check architecture | The fact-checker is an async service layer, not a blocking pipeline stage. Coordinator spawns scouts per-claim, writes annotation files, notifies orchestrator on REFUTED. Planner works with whatever is verified so far. | Blocking stage would delay planning unnecessarily. Async lets verification happen in parallel. REFUTED claims trigger revision, not gates. |
